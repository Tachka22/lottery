# Руководство для разработчиков

## Общая архитектура проекта

Проект строго разделён на слои:

```
Клиент (HTTP)
    |
    V
+-----------------+
|   Controller    |  -> принимает запрос, валидирует параметры, вызывает сервис, возвращает JSON
+--------+--------+
         |
         V
+-----------------+
|    Service      |  -> бизнес-логика (правила, проверки, вызов репозиториев)
+--------+--------+
         |
         V
+-----------------+
|  Repository     |  -> чистая работа с БД (SQL, JDBC)
+--------+--------+
         |
         V
+-----------------+
|   PostgreSQL    |
+-----------------+
```

**Золотое правило:** каждый слой знает только о слое под собой.

- Контроллер **не знает** о репозитории напрямую.
- Сервис **не знает** о контроллере.
- Репозиторий **не содержит** бизнес-логики.

---

## Слой модели (Model)

Модель — это простой Java-объект (POJO) с полями, геттерами и сеттерами.

**Важно:**
- При необходимости нужно настроить маппинг, т.к. нейминг БД и принятой в Java нотации отличается.

---

## Слой репозитория (Repository)

Репозиторий отвечает за **все** SQL-запросы к базе данных.  
Каждый репозиторий - это интерфейс и его реализация.

**Что важно:**
- Всегда закрывайте ресурсы (`try-with-resources`)
- Логируйте ошибки с уровнем `error`
- Не проглатывайте исключения

---

## Слой сервиса (Service)

Сервис содержит **бизнес-логику**. Например:
- Проверка, можно ли начать тираж (только если статус DRAFT).
- Генерация выигрышной комбинации.
- Вычисление победителей.

### Добавление и внедрение новых сервисов (DI)

В нашем проекте для управления зависимостями используется Google Guice. Все связи между интерфейсами и их реализациями настраиваются в классе `LotteryModule`.

Чтобы добавить в систему новый сервис или репозиторий, выполните следующие 3 шага:

#### 1. Создание интерфейса и реализации

Каждый новый компонент должен состоять из двух частей. Реализация должна использовать инъекцию через конструктор.

```java
// 1. Описываем контракт
public interface MyComponent {
    void execute();
}

// 2. Пишем логику
public class MyComponentImpl implements MyComponent {
    @Inject // Сообщаем Guice, что сюда нужно передать зависимости
    public MyComponentImpl(OtherComponent dependency) {
        this.dependency = dependency;
    }
}
```

#### 2. Регистрация в модуле

Любой новый сервис или репозиторий обязательно должен быть зарегистрирован в классе `LotteryModule`. В противном случае приложение не запустится.

```java
public class LotteryModule extends AbstractModule {
    @Override
    protected void configure() {
        // Шаблон для регистрации:
        bind(InterfaceName.class)
            .to(ImplementationName.class)
            .in(Scopes.SINGLETON);
    }
}
```

## Добавление новых эндпоинтов (Routing)

Все маршруты приложения описываются в классе `AppRouter`. Мы используем `ApiBuilder` от Javalin для создания древовидной структуры API.

### 1. Добавление метода в контроллер

Прежде чем регистрировать роут, создайте метод в соответствующем контроллере. Метод обязательно должен принимать `Context`.

```java
public void myNewAction(Context ctx) {
    // 1. Получаем данные (из Body, QueryParams или PathParams)
    // 2. Вызываем сервис
    // 3. Возвращаем результат через ctx.json() или ctx.status()
    ctx.status(200).json(Map.of("message", "Done"));
}
```

### 2. Регистрация в AppRouter

Откройте `AppRouter` и добавьте метод в нужный блок `path`.

```java
public void registerRoutes() {
    path("/draws", () -> {
        // Привязка к GET /draws/info
        get("/info", drawController::myNewAction);

        // Привязка с параметром пути: GET /draws/{id}/status
        get("/{id}/status", drawController::getStatus);
    });
}
```

### 3. Основные правила и глаголы

| Метод       | Аналог в SQL | Когда использовать                |
|-------------|--------------|-----------------------------------|
| `get()`     | SELECT       | Получение данных или списков.     |
| `post()`    | INSERT       | Создание новых сущностей.         |
| `put()` / `patch()` | UPDATE       | Обновление существующих данных.   |
| `delete()`  | DELETE       | Удаление данных.                  |

## Порядок создания нового эндпоинта

1. **Подготовка БД (SQL)**  
   Если для нового эндпоинта нужны новые поля или таблицы, начни с миграции.
   - Создай файл `src/main/resources/db/migration/V2__description.sql`.
   - Напиши SQL (например, `ALTER TABLE draws ADD COLUMN description TEXT`).

2. **Обновление модели (Model)**
   - Добавь необходимые поля в Java-класс сущности.
   - Используй аннотации Lombok (`@Data`), чтобы не писать геттеры/сеттеры вручную.
   - Если добавил статус — обнови соответствующий Enum.

3. **Репозиторий (Repository)**
   - Напиши метод для работы с БД в интерфейсе и его реализации.
   - Interface: добавь метод, например `void updateStatus(int id, Status s)`.
   - Implementation: напиши SQL-запрос, возьми соединение из `dataSource` и выполни `stmt.executeUpdate()`.

4. **Сервис (Service)**
   - Реализуй бизнес-логику.
   - Вызови метод репозитория.
   - Здесь же делай проверки (например, нельзя перевести лотерею в ACTIVE, если в ней нет билетов).
   - **DI:** Убедись, что репозиторий внедрен в сервис через `@Inject`.

5. **Контроллер (Controller)**
   - Создай метод, который свяжет HTTP с сервисом.
   - Используй `ctx.bodyAsClass(DTO.class)` для приема данных.
   - Вызови сервис.
   - Верни ответ через `ctx.json(...)`.

6. **Роутинг (AppRouter)**
   - Зарегистрируй путь.
   - В `AppRouter.java` добавь строку в нужный блок `path`, например:  
     `patch("/{id}/status", drawController::updateStatus);`

7. **Регистрация в DI (LotteryModule)**
   - Если ты создал новые классы (новый Сервис или Репозиторий):  
     Добавь `bind(Interface.class).to(Impl.class).in(Scopes.SINGLETON);` в `LotteryModule`.
