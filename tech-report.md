# Техническая часть 

## Общая архитектура проекта

Проект строго разделён на слои:

```
Клиент (HTTP)
    |
    V
+-----------------+
|   Controller    |  -> принимает запрос, валидирует параметры, вызывает сервис, возвращает JSON
+--------+--------+
         |
         V
+-----------------+
|    Service      |  -> бизнес-логика (правила, проверки, вызов репозиториев)
+--------+--------+
         |
         V
+-----------------+
|  Repository     |  -> чистая работа с БД (SQL, JDBC)
+--------+--------+
         |
         V
+-----------------+
|   PostgreSQL    |
+-----------------+
```

## Зависимости проекта

### Веб-фреймворк и JSON
*   **Javalin (`6.1.3`)**: Легковесный веб-фреймворк для создания REST API.
*   **Jackson Databind (`2.15.2`)**: Основная библиотека для сериализации/десериализации Java-объектов в JSON и обратно.
*   **Jackson JSR310 (`2.15.2`)**: Модуль Jackson для корректной работы с датами и временем (`LocalDateTime`, `ZonedDateTime` и др.).

### Логирование
*   **Logback Classic (`1.5.13`)**: Реализация интерфейса логирования (движок).
*   **SLF4J API (`2.0.9`)**: Абстрактный интерфейс для логирования, позволяющий легко менять реализацию в будущем.

### База данных и миграции
*   **PostgreSQL Driver (`42.7.7`)**: JDBC-драйвер для подключения к базе данных PostgreSQL.
*   **HikariCP (`5.1.0`)**: Высокопроизводительный пул соединений с базой данных (стандарт индустрии).
*   **Flyway Core (`10.20.0`)**: Инструмент для версионирования схемы базы данных (миграции).
*   **Flyway PostgreSQL (`10.20.0`)**: Специфичная поддержка синтаксиса PostgreSQL для Flyway.

### Внедрение зависимостей (DI)
*   **Google Guice (`7.0.0`)**: Легковесный фреймворк для внедрения зависимостей (альтернатива Spring DI). Используется для связывания интерфейсов и реализаций.

### Безопасность и Аутентификация
*   **JJWT API (`0.12.5`)**: Интерфейсы для работы с JWT (JSON Web Tokens).
*   **JJWT Impl & Jackson**: Реализация генерации/валидации токенов и интеграция с Jackson для парсинга payload.

### Почта и Утилиты
*   **Jakarta Mail (`2.0.1`)**: Библиотека для отправки email.

## База данных

Вот описание схемы базы данных в формате Markdown таблиц.

### Схема базы данных Lottery API

#### 1. `users` - Пользователи
Хранит учетные записи пользователей и администраторов.

| Колонка | Тип данных | Ограничения / Описание |
| :--- | :--- | :--- |
| `id` | `SERIAL` | **PK**. Уникальный идентификатор пользователя. |
| `email` | `VARCHAR(100)` | **UNIQUE, NOT NULL**. Email для входа. |
| `password` | `VARCHAR(255)` | **NOT NULL**. Хэш пароля. |
| `role` | `VARCHAR(20)` | **NOT NULL**. Роль: `'USER'` или `'ADMIN'`. |
| `created_at` | `TIMESTAMP` | Дата регистрации (по умолчанию `CURRENT_TIMESTAMP`). |

---

#### 2. `lottery_types` - Типы лотерей
Определяет правила игры (например, "5 из 50", "Кено" и т.д.).

| Колонка | Тип данных | Ограничения / Описание |
| :--- | :--- | :--- |
| `name` | `VARCHAR(30)` | **PK**. Уникальное имя типа (например, `CLASSIC`). |
| `numbers_count` | `INT` | **NOT NULL**. Количество чисел в билете. |
| `min_number` | `INT` | **NOT NULL**. Минимальное возможное число. |
| `max_number` | `INT` | **NOT NULL**. Максимальное возможное число. |
| `has_bonus` | `BOOLEAN` | По умолчанию `FALSE`. Есть ли бонусный шар. |
| `bonus_min` | `INT` | Минимальное значение бонусного шара (если есть). |
| `bonus_max` | `INT` | Максимальное значение бонусного шара (если есть). |
| `description` | `TEXT` | Описание типа лотереи. |

---

#### 3. `draws` - Тиражи
Конкретные розыгрыши, привязанные к типу лотереи.

| Колонка | Тип данных | Ограничения / Описание |
| :--- | :--- | :--- |
| `id` | `SERIAL` | **PK**. Уникальный идентификатор тиража. |
| `name` | `VARCHAR(100)` | **NOT NULL**. Название тиража. |
| `lottery_type_name` | `VARCHAR(30)` | **FK**. Ссылка на `lottery_types.name`. |
| `status` | `VARCHAR(20)` | **NOT NULL**. Статус: `'DRAFT'`, `'ACTIVE'`, `'FINISHED'`, `'CANCELLED'`. |
| `winning_numbers` | `VARCHAR(100)` | Выигрышная комбинация (заполняется при завершении). |
| `winning_bonus` | `INT` | Выигрышный бонусный шар (если применимо). |
| `created_at` | `TIMESTAMP` | Дата создания тиража. |
| `finished_at` | `TIMESTAMP` | Дата завершения тиража. |
| `description` | `TEXT` | Дополнительное описание тиража. |

> **Индексы:**
> *   `idx_draws_lottery_type` на колонке `lottery_type_name` для ускорения фильтрации.

---

#### 4. `tickets` - Билеты
Покупки пользователей в конкретных тиражах.

| Колонка | Тип данных | Ограничения / Описание |
| :--- | :--- | :--- |
| `id` | `SERIAL` | **PK**. Уникальный идентификатор билета. |
| `draw_id` | `INT` | **FK**. Ссылка на `draws.id` (CASCADE DELETE). |
| `user_id` | `INT` | **FK**. Ссылка на `users.id` (CASCADE DELETE). |
| `numbers` | `VARCHAR(100)` | **NOT NULL**. Выбранные пользователем числа. |
| `bonus` | `INT` | Выбранный бонусный шар (если есть). |
| `status` | `VARCHAR(20)` | **NOT NULL**. Статус: `'PENDING'`, `'WIN'`, `'LOSE'`. |
| `created_at` | `TIMESTAMP` | Дата покупки билета. |

---

#### 5. `user_actions` - История действий пользователя
Лог действий пользователей (аудит).

| Колонка | Тип данных | Ограничения / Описание |
| :--- | :--- | :--- |
| `id` | `SERIAL` | **PK**. Уникальный идентификатор записи. |
| `user_id` | `INT` | **FK**. Ссылка на `users.id` (CASCADE DELETE). |
| `action_type` | `VARCHAR(50)` | **NOT NULL**. Тип действия (например, `TICKET_PURCHASE`). |
| `details` | `JSONB` | Дополнительные данные в формате JSON. |
| `created_at` | `TIMESTAMP` | Время совершения действия. |

---

## Реализованный дополнительный функционал

В рамках разработки Lottery API были успешно реализованы следующие ключевые требования:

### 1. История операций пользователя (User Action History)
*   **Реализация:** Создан механизм сквозного аудита действий пользователей.
*   **Архитектура:** Использован паттерн **Event Bus** (шина событий). Сервисы публикуют события при выполнении ключевых действий (покупка билета, вход в систему), а отдельный `AuditService` подписывается на них и сохраняет данные в таблицу `user_actions`.
*   **Доступ:** Предоставлен REST-эндпоинт `GET /users/{userId}/history`, позволяющий пользователю просматривать хронологию своих действий с защитой от доступа к чужим данным (проверка прав на уровне контроллера).
*   **Масштабируемость:** Позволяет легко добавить новых подписчиков и использовать брокеры
* 
### 2. Экспорт отчетов по завершенным тиражам
*   **Форматы:** Реализована гибкая выгрузка данных в форматах **JSON** и **CSV**.
*   **Эндпоинт:** `GET /reports/draws/{drawId}`.
*   **Детализация:** Отчет содержит полную информацию о тираже (выигрышные числа, дата завершения) и список всех участвовавших билетов с их статусами.

### 3. Обработка отмены тиража (Cancellation Logic)
*   **Целостность данных:** При отмене тиража система автоматически обновляет статусы всех связанных билетов на `CANCELLED` (или удаляет их, в зависимости от бизнес-правила, в текущей реализации используется каскадное обновление/удаление через БД).
*   **Валидация:** Запрещена отмена уже завершенных (`FINISHED`) тиражей для сохранения исторической достоверности результатов.

### 4. Уведомление пользователей по Email
*   **Интеграция:** Подключена библиотека **Jakarta Mail** для отправки электронных писем.
*   **Архитектура:** Использован паттерн стратегия.
*   **Асинхронность:** Процесс отправки писем не блокирует основной поток завершения тиража.

### 5. Поддержка множественных типов лотерей (Multi-Type Support)
*   **Гибкость:** Система не жестко привязана к одному формату игры. Реализована сущность `LotteryType` (таблица `lottery_types`).
*   **Параметризация:** Администратор может создавать новые типы лотерей с различными правилами:
    *   Количество выбираемых чисел (`numbersCount`).
    *   Диапазон чисел (`minNumber`, `maxNumber`).
    *   Наличие и параметры бонусного шара (`hasBonus`, `bonusMin`, `bonusMax`).
*   **Масштабируемость:** Каждый тираж (`Draw`) строго типизирован и ссылается на конкретный `LotteryType`, что позволяет валидировать билеты и генерировать выигрышные комбинации в соответствии с правилами конкретного типа игры.
